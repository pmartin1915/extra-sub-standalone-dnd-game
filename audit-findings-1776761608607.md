# Audit Findings

### Audit Report: sandbox-dnd-game

**Date:** 2026-04-21 (UTC)
**Project:** sandbox-dnd-game
**Status:** Pre-implementation (planning phase)

---

### 1. Audit Summary

This audit was performed on the project's planning and governance documents, as no source code has been written yet. The project has a clear and ambitious charter with well-defined goals, particularly the "Teaching Toggle" feature and the planned integration with the `worldbuilder` project.

The overall structure is sound, but there are several strategic and procedural risks that should be addressed before implementation begins. The findings focus on clarifying ambiguities in the charter, resolving potential process deadlocks, and recommending best practices for project initialization to ensure a smooth transition from planning to development.

---

### 2. Findings & Recommendations

#### Finding 1: Ambiguity in Intellectual Property Guardrails
**Severity:** `Medium`

**Description:**
The `CLAUDE.md` charter contains conflicting goals. It calls for "Real D&D-style mechanics" but also includes a hard constraint to "NEVER...Copying D&D copyrighted content verbatim". The line between a "mechanic" (not copyrightable) and "expression" (copyrightable) is notoriously fine. Terms like "Armor Class" or "d20 roll" are generally considered mechanics, but specific class names, monster names (e.g., "Beholder"), and spell descriptions are protected intellectual property. This ambiguity creates a risk of either accidental infringement or unnecessary deviation from familiar concepts, harming the user experience.

**Recommendation:**
Before design work begins, create a `design/IP_GUIDELINES.md` document. This document should explicitly define the project's stance on what can and cannot be used.

-   **Define a "safe list"** of common TTRPG terms (e.g., Ability Scores, Hit Points, d20 System).
-   **Establish a clear process** for creating original names for classes, spells, and monsters that are evocative of the D&D experience without using trademarked or copyrighted terms.
-   This task should be assigned via the `research` keyword in `DISPATCH.md` to formalize the project's approach.

#### Finding 2: Procedural Deadlock on Project Initialization
**Severity:** `Medium`

**Description:**
`ai/STATE.md` correctly identifies the next step as creating the project structure (`package.json`, `tsconfig.json`). However, `CLAUDE.md` and `DISPATCH.md` explicitly forbid `npm install` without manual approval from "Perry". This creates a procedural deadlock where the autonomous agent cannot initialize the project or install essential development dependencies like TypeScript or a testing framework.

**Recommendation:**
To unblock development, one of the following actions should be taken:
1.  **Pre-approval of Core Dependencies:** The project owner ("Perry") should pre-approve a minimal set of essential `devDependencies` (e.g., `typescript`, `ts-node`, `@types/node`, and a testing framework like `jest` or `vitest`).
2.  **Create a Seed `package.json`:** The owner should create and commit a `package.json` file with these approved dependencies. The autonomous agent can then be permitted to run `npm install` to sync with the committed file, without making changes to it.

#### Finding 3: Late-Phase Planning for Lore Adapter Interface
**Severity:** `Low`

**Description:**
The `ROADMAP.md` places the "Lore adapter interface for worldbuilder integration" in Phase 4. While the full implementation belongs there, designing the interface (i.e., the TypeScript `interface` or `type` definitions for lore objects like `NPC`, `Region`, `Faction`) should occur much earlier. If the core engine is built without knowing the shape of the data it will eventually consume, significant refactoring will be required during Phase 4 integration.

**Recommendation:**
Move the *design* of the lore adapter interface to Phase 2 ("Engine Foundation").
-   Create a file like `src/interfaces/lore.ts`.
-   Define placeholder TypeScript interfaces for all expected data structures from the `worldbuilder` project.
-   Build the game engine against these interfaces from the beginning. This will make the final integration in Phase 4 a simple matter of implementing the adapter, not re-architecting the engine.

---

#### Finding 4: Lack of Defined Tooling and Quality Standards
**Severity:** `Low`

**Description:**
The project is specified to use TypeScript, but there is no mention of code quality tooling such as linters (ESLint) or code formatters (Prettier). Establishing these standards at the project's inception is critical for maintaining code consistency and preventing common errors, especially in an autonomously managed project.

**Recommendation:**
As part of the project initialization (see Finding 2), include and configure ESLint and Prettier. Create configuration files (`.eslintrc.json`, `.prettierrc.json`) in the root directory. This ensures all contributed code, regardless of the agent that wrote it, adheres to the same quality standard.

---

#### Finding 5: Vague Definition of "Game Balance"
**Severity:** `Informational`

**Description:**
The charter requires "Balanced encounters" and "actual challenge ratings". Game balance is highly subjective and difficult to achieve without clear, measurable goals. The roadmap does not include a specific task to define what "balance" means for this project.

**Recommendation:**
Add a dedicated task to Phase 1: "[ ] Define and document the game's balance framework". This should be a `research` task that results in a design document (`design/BALANCE.md`). The document should answer questions like:
-   What is the target difficulty? (e.g., "A party of 4 level-appropriate characters should have a ~65% chance of winning an average encounter with ~25% resource depletion.")
-   How will Challenge Rating (CR) be calculated for custom monsters?
-   How will encounter difficulty be calculated based on party size, level, and monster CR?
-   This provides a concrete target for the implementation and testing phases.
